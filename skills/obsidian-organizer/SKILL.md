# Obsidian 문서 정리 스킬

이 스킬은 Obsidian 볼트에서 정리되지 않은 문서를 카테고리별로 분류하고, 파일명에 숫자 접두어를 부여하여 일관된 구조로 만드는 작업을 수행한다.

## 핵심 원칙

이 스킬이 지키는 가장 중요한 원칙은 **사용자의 데이터를 절대 잃지 않는 것**이다.

1. **백업 우선**: 파일을 수정·이동·삭제하기 전에 반드시 원본 콘텐츠를 메모리에 보관한다.
2. **검증 필수**: 새 파일 생성 후, 내용이 원본과 일치하는지 확인한 뒤에만 기존 파일을 삭제한다.
3. **기존 구조 존중**: 이미 정리된 폴더 구조는 건드리지 않고, 정리가 필요한 문서만 적절한 위치에 배치한다.
4. **사전 승인**: 실행 전 반드시 사용자의 승인을 받는다 — 문서 이동은 되돌리기 어려운 작업이기 때문이다.

## 사용자 상호작용 원칙

사용자에게 질문하거나 선택을 요청할 때는 **반드시 AskUserQuestion 도구를 사용**한다. 일반 텍스트로 질문하지 않는다. 이렇게 하면 사용자가 선택지를 명확히 보고 빠르게 응답할 수 있다.

## 워크플로우

### 1단계: 정리 범위 파악

사용자가 정리 범위를 지정했는지 확인한다.

- **범위가 지정된 경우** (예: "독서 폴더 정리해줘", "루트에 있는 파일들 정리해줘"): 해당 경로만 스캔한다.
- **범위가 지정되지 않은 경우**: 사용자에게 정리할 범위를 물어본다. 전체 볼트를 원하면 전체를 스캔한다.

범위 파악은 **점진적으로 좁혀가는 방식**으로 진행한다. 사용자가 큰 범위(예: 최상위 폴더)를 선택하면, 그 안의 하위 폴더 구조와 미정리 파일 수를 함께 보여주고 세부 범위를 한 번 더 확인한다. 이렇게 하는 이유는 사용자가 "개발 폴더 정리해줘"라고 해도, 실제로는 특정 하위 폴더만 정리하고 싶을 수 있기 때문이다.

예시:
```
개발/ 폴더 안의 구조입니다:
├── 01. 기초 & CS/       (파일 N개)
├── 02. 프론트/          (파일 N개)
├── 03. 백엔드/          (파일 N개)
├── ...
└── 루트 파일: 1개 (개발 공수 산정 WBS 작성 방법.md)

전체를 정리할까요, 특정 하위 폴더만 정리할까요?
```

Obsidian MCP 도구(`obsidian_list_files_in_dir` 또는 `obsidian_list_files_in_vault`)를 사용해 대상 범위의 파일과 폴더 목록을 가져온다.

### 2단계: 현황 분석

스캔 결과를 바탕으로 다음을 파악한다:

- **이미 정리된 영역**: 폴더 안에 체계적으로 배치된 문서. 이들은 유지한다.
- **정리가 필요한 문서**: 루트에 흩어져 있거나, 성격에 맞지 않는 위치에 있는 문서.

정리가 필요한 문서의 내용을 파악하기 위해 `obsidian_get_file_contents`로 문서 앞부분을 읽는다. 제목과 내용을 기반으로 각 문서의 성격을 판단한다.

- **중복 문서 감지**: 제목이나 내용이 유사한 문서가 발견되면, **반드시 모든 의심 문서의 내용을 `obsidian_get_file_contents` 또는 `obsidian_batch_get_file_contents`로 직접 읽어서 확인한다.** 추측이나 "확인 필요"로 넘기지 않는다. 모든 문서의 내용을 확인한 후, 각 문서의 실제 내용 요약(분량, 다루는 주제, 관점 차이)을 사용자에게 제시하고 AskUserQuestion으로 처리 방법을 물어본다:
  - **하나로 병합**: 내용을 합쳐 하나의 문서로 통합하고 나머지 삭제
  - **모두 유지**: 각각 다른 관점이므로 별도 문서로 보존
  - **특정 문서만 유지**: 사용자가 유지할 문서를 선택

### 3단계: 카테고리 결정

분류 기준은 다음 순서로 적용한다:

1. **사용자가 사전 정의한 카테고리가 있는 경우**: 해당 카테고리를 우선 사용한다. 사용자에게 카테고리 목록이 있는지 먼저 물어본다.
2. **기존 폴더 구조에서 카테고리를 유추**: 이미 볼트에 존재하는 폴더명을 카테고리로 활용한다.
3. **새 카테고리 제안**: 기존 카테고리에 맞지 않는 문서가 여러 개 모이면, 새 카테고리를 만들어 제안한다.

특정 카테고리로 분류하기 어려운 문서는 루트 폴더(또는 지정된 범위의 최상위)에 그대로 둔다. 억지로 분류하지 않는다.

### 4단계: 정리 계획 수립 및 제시

다음 규칙에 따라 정리 계획을 세운다:

#### 파일명 접두어 규칙

기본 형식은 `01. 문서제목.md`이다 (번호 + 점 + 공백 + 제목). 단, 기존 폴더에 이미 다른 접두어 스타일이 사용되고 있다면 그 스타일을 따른다. 한 폴더 안에서 접두어 스타일이 섞이면 오히려 더 지저분해지기 때문이다.

- 같은 폴더 내 문서들은 01부터 순차적으로 번호를 부여한다.
- **정리 완료 후 해당 폴더의 전체 파일 번호를 01부터 재정렬한다.** 빈 번호 없이 연속된 번호로 다시 매긴다. 기존 파일의 상대적 순서는 유지하되, 새로 추가된 파일은 내용의 연관성에 따라 적절한 위치에 배치한다.
- 기존 파일명의 의미 있는 부분은 보존한다.

#### 폴더 규칙

새로 만드는 폴더에는 숫자 접두어를 붙이지 않는다. 단, 기존 폴더에 이미 번호가 붙어있는 경우(예: `01. 경력기술서/`)는 그대로 유지한다. 기존 구조를 존중하는 것이 일관성을 깨뜨리는 것보다 중요하기 때문이다.

- 새 폴더를 만들 때, 같은 레벨의 기존 폴더들이 번호 접두어를 사용하고 있다면, 새 폴더에도 같은 스타일로 번호를 부여한다 (예: 기존에 `01. 경력기술서/`, `02. 자기소개서/`가 있으면 새 폴더는 `05. 취업 전략/`).
- 폴더명은 카테고리의 성격을 명확히 드러내는 간결한 한글 또는 영문으로 한다.
- 최대 폴더 깊이는 5단계를 넘지 않는다.

#### 기존 구조 처리

이 스킬의 핵심 철학은 "기존 구조 유지 + 미정리 문서만 배치"이다.

- 이미 정리된 폴더와 그 안의 파일은 유지한다.
- 기존 폴더의 접두어 스타일(번호 형식, 구분자)을 파악하고, 새 파일/폴더 추가 시 같은 스타일을 따른다.
- 기존 폴더 안의 파일에 접두어가 없으면, 접두어를 추가하는 것을 제안할 수 있다 (사용자 확인 후).

#### 계획 제시 형식

사용자에게 트리 형태로 변경 계획을 보여준다. 변경 사항은 명확히 표시한다:

```
정리 계획:
/대상폴더/
├── 개발/                        ← 기존 유지
│   ├── 01. 리액트 훅 정리.md     ← 접두어 추가
│   └── 02. API 설계 원칙.md      ← 이동 + 접두어
├── 업무/                         ← 새 폴더
│   ├── 01. 회의록 2025-01-15.md  ← 이동
│   └── 02. 주간회고 3주차.md     ← 이동
└── 이력서 초안.md               ← 루트 유지 (분류 불명확)
```

변경 요약도 함께 제시한다:
- 새로 생성되는 폴더: N개
- 이동되는 문서: N개
- 이름이 변경되는 문서: N개
- 루트에 유지되는 문서: N개

### 5단계: 사용자 승인

**반드시 사용자의 명시적 승인을 받은 후에만 실행한다.**

사용자가 수정을 요청하면 (카테고리 변경, 문서 재배치, 번호 조정 등) 계획을 수정하고 다시 제시한다.

### 6단계: 실행

승인을 받으면 Obsidian MCP 도구를 사용해 변경을 적용한다.

#### ⚠️ 데이터 손실 방지 경고

실행 시작 전, 사용자에게 다음 경고를 **반드시** 표시한다:

> **주의**: 파일 이동/이름변경 작업은 되돌리기 어렵습니다.
> - **Git 사용자**: `git add -A && git commit -m "backup: 정리 작업 전"` 으로 백업하셨나요?
> - **Git 미사용자**: 대상 폴더를 별도로 복사해두시는 것을 권장합니다.
> - 계속 진행할까요?

사용자가 확인한 후에만 실행을 시작한다.

#### 실행 원칙: 한 파일씩 순차 처리

**절대 Task 에이전트에게 파일 이동/이름변경을 위임하지 않는다.** 반드시 메인 에이전트가 직접 한 파일씩 처리한다. Task 에이전트는 파일 내용을 플레이스홀더로 대체하는 등 예측 불가능한 동작을 할 수 있기 때문이다.

#### 실행 순서 (파일 이동)

각 파일에 대해 다음 순서를 **엄격히** 따른다:

1. **읽기**: `obsidian_get_file_contents`로 원본 내용 전체를 읽어 메모리에 보관한다.
2. **읽기 검증**: 읽은 내용이 비어있거나 에러(40400 등)가 발생하면, **해당 파일은 건너뛰고** 사용자에게 알린다. 빈 콘텐츠로 덮어쓰지 않는다.
3. **생성**: `obsidian_append_content`로 새 경로에 원본 내용을 작성한다.
4. **생성 검증**: `obsidian_get_file_contents`로 새 파일을 읽어 원본과 **내용이 일치하는지** 확인한다.
5. **삭제**: 검증이 통과한 경우에만 `obsidian_delete_file`로 기존 파일을 삭제한다.

> **중요**: 3→4→5 단계에서 검증 없이 삭제하면 데이터가 손실된다. 반드시 4단계 검증을 수행한다.

#### 실행 순서 (이름변경/재번호)

파일 이름만 변경하는 경우에도 동일한 원칙을 적용한다:

1. **읽기**: 원본 내용 전체를 읽는다.
2. **읽기 검증**: 내용이 비어있거나 에러면 건너뛴다.
3. **생성**: 새 이름으로 파일을 생성하고 원본 내용을 작성한다.
4. **생성 검증**: 새 파일 내용이 원본과 일치하는지 확인한다.
5. **삭제**: 검증 통과 후 기존 파일을 삭제한다.

#### 특수문자 파일명 처리

파일명에 `?`, `#`, `%` 등 특수문자가 포함된 경우 Obsidian MCP에서 읽기/삭제가 실패할 수 있다. 이 경우:

1. 사용자에게 해당 파일을 **Obsidian 앱에서 직접** 이름을 변경하도록 안내한다.
2. 또는 해당 파일은 건너뛰고 나머지 파일만 처리한다.
3. **절대로** 읽지 못한 파일을 빈 내용이나 플레이스홀더로 대체하지 않는다.

#### 대량 작업 시 중간 체크포인트

10개 이상의 파일을 처리하는 경우, 5개마다 진행 상황을 사용자에게 보고하고 문제가 없는지 확인한다.

#### 폴더 생성

새 폴더가 필요한 경우, 해당 폴더에 임시 문서를 생성하여 폴더를 만든다. 모든 작업 완료 후 임시 문서를 삭제한다.

### 7단계: 결과 보고

모든 작업이 완료되면 결과를 요약한다:

```
정리 완료!
- 새로 생성된 폴더: 2개 (업무/, 독서/)
- 이동된 문서: 4개
- 이름 변경된 문서: 3개
- 루트에 유지된 문서: 1개 (이력서 초안.md)
```

최종 폴더 구조도 트리로 보여준다.

## 스킬 자기 업데이트

이 스킬은 사용자의 취향 변화를 반영하여 자기 자신을 업데이트할 수 있다.
자기 업데이트 프로세스는 공통 가이드를 따른다: `shared-references/self-update.md` 참조.

변경 가능 항목:
- 접두어 형식 (번호 스타일, 구분자)
- 폴더 명명 규칙 (접두어 유무, 언어)
- 카테고리 분류 기준 및 사전 정의 카테고리 목록
- 폴더 깊이 제한
- 기존 구조 처리 방식
- 계획 제시 형식

변경 불가 항목:
- 실행 전 사용자 승인 필수 원칙
- 실행 전 백업 경고 필수 원칙
- 읽기→생성→검증→삭제 순서 원칙
- Task 에이전트 위임 금지 원칙
- 기존 문서 내용 보존 원칙
- `.obsidian/` 폴더 보호

## 주의사항

### 데이터 안전
- **읽기 실패한 파일은 절대 덮어쓰지 않는다.** 40400 에러, 빈 응답, 인코딩 오류 등이 발생하면 해당 파일은 건너뛰고 사용자에게 보고한다.
- **Task 에이전트에게 파일 이동/삭제/이름변경을 위임하지 않는다.** 반드시 메인 에이전트가 직접 처리한다. 이는 이전에 Task 에이전트가 원본 콘텐츠를 플레이스홀더로 대체하여 대량 데이터 손실이 발생한 사고의 재발을 방지하기 위함이다.
- **파일명에 `?`, `#`, `%` 등 특수문자**가 포함된 파일은 Obsidian MCP에서 정상 작동하지 않을 수 있다. 이런 파일은 사용자에게 Obsidian 앱에서 직접 이름을 변경하도록 안내한다.

### 일반
- Obsidian의 내부 링크(`[[문서명]]`)가 있는 경우, 문서 이동 시 링크가 깨질 수 있다. 이를 사용자에게 미리 알린다. Obsidian 자체의 자동 링크 업데이트 기능이 있으므로, Obsidian 앱이 열려 있는 상태에서 작업하면 링크가 자동으로 갱신될 수 있다.
- `.obsidian/` 폴더나 설정 파일은 절대 건드리지 않는다.
- 첨부파일(이미지, PDF 등)은 이 스킬의 범위에 포함하지 않는다. 마크다운 문서(.md)만 대상으로 한다.
- 한 번에 너무 많은 문서를 이동하면 실수 가능성이 높아진다. 20개 이상의 문서를 이동해야 하는 경우, 카테고리별로 나눠서 단계적으로 진행하고 중간중간 사용자에게 확인받는다.
